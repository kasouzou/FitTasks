# `onClick` と `onSave` の定義箇所と仕組みの解説

## 概要
Android（Jetpack Compose）におけるイベントの定義は、**「器（引数）」** と **「中身（実装）」** に分かれています。ここでは、保存ボタンを例に、Kotlin 特有の書き方や Dart との比較を交えて解説します。

---

## 1. 基礎知識： `Unit` と `(TaskGroup) -> Unit`

コード内で頻出するこの書き方の意味を紐解きます。

### `Unit` ってなに？
結論から言うと、**Dart や Java の `void` とほぼ同じ**です。「何も返さない（戻り値がない）」ことを表しています。

*   **なぜ `void` ではないのか？**: Kotlin の `Unit` は「Unit という型のインスタンスが一つだけ存在する」という扱いです。すべての関数を同じルール（必ず何かを返す）で扱うための工夫です。

### `(TaskGroup) -> Unit` の読み解き方
これは **「関数そのものを引数として受け取る（高階関数）」** ための型定義です。

*   **`(TaskGroup)`**: 引数に `TaskGroup` 型を1つ受け取り、
*   **`->`**: それを実行して、
*   **`Unit`**: 何も返さない。

つまり、**「タスクグループを渡すと、何か処理（保存など）をしてくれる関数」** という宣言です。

### Dart との比較
Dart で書くなら、以下の書き方と全く同じ意味になります。
```dart
final void Function(TaskGroup) onSave;
```

---

## 2. 定義と実行の連鎖（バケツリレー）

「関数の形をした空箱」を定義しておいて、後から「中身」を流し込むという仕組みを、宣言時と実行時に分けて整理します。

### ① 宣言時（契約）： `TaskEditScreen.kt`
「この画面を使うなら、保存用の関数を渡してね」という**契約**です。

```kotlin
@Composable
fun TaskEditScreen(
    ...,
    onSave: (TaskGroup) -> Unit // 器（引数の型）の定義
) { ... }
```
*   **立ち位置**: 画面（子）は「どうやって保存するか」を知りません。ただ「この型に沿った関数が外からやってくる」と信じている状態です。

### ② 実行時（トリガー）： `TaskEditScreen.kt` 内のボタン
実際にボタンが押された瞬間に、外から預かっていた関数（空箱）にデータを詰め込んで実行します。

```kotlin
ExtendedFloatingActionButton(
    onClick = { // Androidフレームワークが定義した「タップ検知」の器
        onSave(TaskGroup(...)) // 外から届いた関数にデータを渡して動かす！
    }
)
```

### ③ 中身の注入（実装）： `MainActivity.kt`
親が具体的な「保存のやり方」を定義し、子の引数に流し込みます。

```kotlin
TaskEditScreen(
    onSave = { group -> // ここで「onSave の具体的な中身」を実装
        saveTaskGroupUseCase(group) // ユースケースの実行
    }
)
```

---

## 3. 役割分担のまとめ

| 段階 | 定義場所 | 役割 | 状態 |
| :--- | :--- | :--- | :--- |
| **フレームワーク** | Material3 ライブラリ | `onClick` という「タップ検知」の器を提供。 | 土台 |
| **宣言（子）** | `TaskEditScreen.kt` | `onSave` という「保存の意図」を伝える器を定義。 | 契約（空箱） |
| **実装（親）** | `MainActivity.kt` | `onSave` の中身（UseCase呼び出し）を定義。 | 中身（具） |
| **実行（発火）** | `TaskEditScreen.kt` | ボタン押下時に `onSave(data)` を呼び出す。 | トリガー |

---

## 4. なぜこの書き方が重要か？（論理的なつながり）

この構造は、**「ステートホイスティング（状態の引き上げ）」** と呼ばれる重要な設計パターンです。

1.  **疎結合**: UI（子）は保存の仕組み（DBなのかサーバーなのか）を知らなくて良いため、再利用しやすくなります。
2.  **UseCase との親和性**: UseCase は `invoke` を持つオブジェクトであり、`(TaskGroup) -> Unit` という型と互換性があります。そのため、`onSave = saveTaskGroupUseCase` と直接代入できるという強力な連携が可能になります。

## 結論
*   **宣言時**: `変数名: (型) -> Unit` で関数の形を決める。
*   **実行時**: `変数名(実際のデータ)` でイベントを親へ跳ね返す。

この「関数を変数として持ち運ぶ」仕組みこそが、モダンなアプリ開発において UI とビジネスロジックを繋ぐ最強の接着剤となります。
