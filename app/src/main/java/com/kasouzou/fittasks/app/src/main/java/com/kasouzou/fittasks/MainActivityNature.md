# MainActivityの性質と役割のまとめ

## 概要
本プロジェクトにおける `MainActivity` は、単なるアプリのエントリポイント（入り口）ではなく、アプリ全体の構成、依存関係、画面遷移を管理する **「司令塔（オーケストレーター）」** としての極めて重要な役割を担っています。

## 主要な5つの性質

### 1. 構成ルート (Composition Root)
アプリが正しく動作するために必要な「部品（インスタンス）」をすべて組み立て、保持する場所です。
- **データベース**: `Room.databaseBuilder` でDBを構築。
- **リポジトリ**: `RoomTaskRepository`（具象クラス）を生成。
- **ユースケース**: 保存用などのユースケースを実体化。

### 2. 手動依存注入 (Manual DI) のハブ
DIライブラリ（Hilt等）を使わずに、各コンポーネントへ依存関係を流し込む「起点」になっています。
- **ViewModelへの供給**: `ViewModelFactory` にリポジトリを渡し、ViewModelが必要な「材料」を供給。
- **Composableへの供給**: 各画面に対し、引数経由で必要なデータや関数（コールバック）を渡す。

### 3. ナビゲーション（画面遷移）の管理者
「いまどの画面を表示すべきか」という状態を管理し、画面を切り替える役割です。
- `currentScreen` という状態（`Screen` シールドインターフェース）を持ち、`when` 文を使って `TaskList`, `TaskEdit`, `Timer` 画面を動的に出し分けます。

### 4. ビジネスロジックの最終実行役
UI（各Composable画面）から上がってきた「ユーザーの意図（保存したい、戻りたい）」に対し、具体的なアクションを実行します。
- **例**: `onSave` の中身として、コルーチン（`launch`）を起動し、ユースケースを呼び出して保存処理を完了させます。

### 5. OSとUIの仲介者 (Infrastructural Bridge)
Android OS 特有のシステム機能と、Compose によるモダンな UI を結びつけています。
- `enableEdgeToEdge()` による全画面描画の設定。
- `MobileAds.initialize()` による広告機能（AdMob）の初期化。
- `FitTasksTheme` によるアプリ全体の共通デザインの適用。

---

## `MainActivity` の立ち位置のイメージ

- **UI画面 (TaskListScreenなど)**: 専門の「職人」。ボタンの配置や入力の受付に専念する。
- **ViewModel**: 「在庫管理者」。画面に必要なデータを保持し、整理して職人に渡す。
- **MainActivity**: **「現場監督」**。職人を手配し（画面生成）、必要な道具（Repository/UseCase）を渡し、全体の工程（画面遷移）を管理する。

## 結論
`MainActivity` がすべての依存関係を把握し、一連の流れ（保存→画面戻しなど）を統括することで、各UI画面は「特定の保存方法」や「複雑なデータの出所」を知ることなく、純粋な表示ロジックに集中できています。これが、このプロジェクトにおける疎結合なクリーンアーキテクチャを支える基盤となっています。
